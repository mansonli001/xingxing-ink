### 二、9 天里我学到了什么（凝练 5 条）

#### ✅ 1. 速度比完美重要（Vibe Coding 的真谛）

35 版 / 9 天 = 平均 1 天 4 版。如果当时纠结"先把架构想清楚再开始"，今天连 BP 都生不出来。

**给未来的提醒**：能跑就先跑。完美主义是慢性死亡。

#### ✅ 2. 外部专家意见比想象中重要

v0.7.12.0 那次外部专家评测吸收，22 条意见里有 3 个救命洞察（qProgress 虚高 / BP 门槛太低 / 12-18 轮目标错了）。如果当时盲推自己的 plan，会上线一个数据虚高 + 必产空 BP 的劣化版。

**给未来的提醒**：自尊心 < 用户体验。专家说得对就吸收，不要怕承认自己的 plan 错。

#### ✅ 3. 真痛点 > 设计漂亮

v0.7.12.1 设计了三按钮弹窗（继续聊 / 硬要出 / 关闭），看起来挺周到。结果用户实测「点了完全没反应」+「就算弹窗出来，让我离开去主对话流找追问也割裂」。v0.7.12.2 直接重写为表单版——把追问拉进弹窗里答，体验从 3 步压缩成 1 步。

**给未来的提醒**：用户报"我用着别扭"时，先怀疑设计本身，不是用户不会用。

#### ✅ 4. 文档和代码同等重要（双 repo 文档同步策略）

每个里程碑版本都做"5 份主 repo 文档 + 2 份 mind repo 文档"的同步校对。今天写 v1.0 时，能直接拉 ROADMAP / CHANGELOG / RETROSPECTIVE 看完整轨迹——这就是文档的复利。

**给未来的提醒**：写代码 30min 配 10min 写文档。未来的你会感谢现在的你。

#### ✅ 5. 18 条铁律比单次成功更重要

v0.7.12.0 → v0.7.12.2 累计沉淀的 18 条铁律（写在 RETROSPECTIVE.md），每一条都是踩坑换来的。这些铁律才是 vibe coding 的真正复利——下个项目直接复用。

**给未来的提醒**：每次踩坑都写下"未来不再踩"的提醒。这些提醒比代码本身更值钱。

## v0.7.12.2 复盘 · 「BP 弹窗内嵌追问表单（v0.7.12.1 重构）」

**版本号**：v0.7.12.2
**上线日期**：2026-05-18（周五收档后第一个工作日）
**工作量**：约 1.2h（plan 预估 45-60min · 实际略超）
**commit 数**：1 个集中 commit

### 一、本次想解决什么

v0.7.12.1 上线后用户实测踩坑：「我聊了 11 轮，点「出诊断书」按钮 → 屏幕完全没反应 / 没弹窗」。

**双重问题**：
1. **bug 层面**：v0.7.12.1 弹窗在用户浏览器没出来（推测浏览器缓存旧 v0.7.12.0 JS / 或 z-index 被遮）
2. **设计层面**：v0.7.12.1 设计本身有缺陷——拦下后让醒醒在**主对话流**追问，用户要"看到追问 → 答 → 再点出 BP"三步操作，体验割裂

**用户原话**：「直接弹出需要回答的问题作为补全，最好就在弹出的窗口填写吧」

### 二、踩了哪些坑

#### 🟡 坑 1：v0.7.12.1 设计时没考虑"用户在哪里答"

**经过**：v0.7.12.1 设计时把"姐姐追问"放在主对话流（用 `injectAssistantMessage`）。但用户实际期望是"问题就在弹窗里答"——**用户不想离开弹窗**。

**根因**：设计时把"追问"和"对话"绑定，没意识到对用户来说追问是**为出 BP 服务的填空题**，不是**新的对话回合**。

**给未来的提醒**：
- ⚠️ 任何"为下一步操作收集信息"的交互，**信息收集 + 触发下一步必须在同一容器内闭环**
- ⚠️ 不要让用户"看完 A → 离开容器 → 去 B 答 → 回到 A 触发"——这是三步操作
- ⚠️ 弹窗 = 用户专注解决一件事的容器，能在弹窗里闭环就别让用户跳出去

#### 🟡 坑 2：v0.7.12.1 弹窗"完全没反应"问题没真正定位

**经过**：用户报「点了完全没反应」。我去线上测 API 完全正常 / 代码逻辑也对 / tsc/build 都过。**根因不明**——可能是浏览器缓存 v0.7.12.0 旧 JS、或 z-index 被某个老 modal 遮住、或弹窗被超出屏幕滚动出去了。

**修复方式**：v0.7.12.2 整个 BPGateDialog 重写（包含尺寸、z-index、scroll 行为全升级），等同于强制刷新组件——**实际是用"重写"绕过 bug，没真正定位**。

**给未来的提醒**：
- ⚠️ "完全没反应"这种 bug 必须**让用户开浏览器 DevTools 看 Network + Console**，盲改是赌运气
- ⚠️ 但实战中"绕过 + 重做"有时比"定位 + 修"更快——前提是你确信新方案更好
- ⚠️ 重做后**真机实测**必须排上日程，否则可能再踩同样的坑

#### 🟡 坑 3：bridge API 接口大改但没标 BREAKING CHANGE

**经过**：bridge API 从 `{ ok, content: string, fallback }` 改成 `{ ok, questions: [...], source }`。这是 BREAKING CHANGE，但 v0.7.12.1 的旧调用方 `handleContinueChatFromGate` 也被同步删了，所以没暴露 bug。

**给未来的提醒**：
- ⚠️ API 接口改返回结构时，**先在 CHANGELOG 标 BREAKING**，再去改调用方
- ⚠️ 同 commit 改 API + 改调用方虽然没事，但 git bisect 时会很难定位
- ⚠️ 大改 API 时考虑加版本（如 `?v=2`）保留旧路径一段时间

### 三、对的决策

#### ✅ 决策 1：bridge API 失败时返 ok=true + source="fallback"

**做法**：LLM 失败不返 502，返 200 + 静态 questions

**为什么对**：
- 弹窗永远有内容可填，不需要前端写一堆 if-else 处理 502
- `source` 字段方便后续埋点分析 LLM 命中率

#### ✅ 决策 2：12 题 × 三档 = 36 条静态兜底 prompt

**做法**：写死在 bridge route 里，不需 LLM 也能用

**为什么对**：
- LLM 失败成本归零
- 三档差异化（casual 嫌弃宠溺 / rational 理性 / scathing 精准）100% 保证
- 文件 +200 行但用户体验托底

#### ✅ 决策 3：表单按钮文案「（没填也行）写诊断书」

**做法**：用户没填任何答案时按钮文案变成「（没填也行）写诊断书」（hasAnyAnswer 判断）

**为什么对**：
- 不强制用户答（保留主权）
- 但用文案暗示"姐知道你没填，但还是给你写"——保持调性
- 没填 → 走 force=true → 报告标 forced 加水印——闭环

### 四、给未来的提醒（铁律 17→18 条）

18. **"为下一步操作收集信息"的交互必须在同一容器内闭环** —— v0.7.12.1 教训

## v0.7.12.1 复盘 · 「BP 拦截 = 触发姐姐主动追问」

**版本号**：v0.7.12.1
**上线日期**：2026-05-15（v0.7.12.0 当天即追加补丁）
**工作量**：约 2.5h（plan 预估 2-3h · 实际接近 3h）
**commit 数**：1 个集中 commit（含代码 + 文档）

### 一、本次想解决什么

v0.7.12.0 上线后用户**当场真机踩坑**：聊够 11 轮但被门槛拦下，弹错误「再补 2 块拼图」。

**用户原话**：「我不知道怎么补？这个很有问题，不能这么拦截？如果拦截应该让姐姐直接问出这个问题。」

### 二、踩了哪些坑

#### 🔴 坑 1：v0.7.12.0 设计时只想了"门槛"，没想"被拦后用户怎么办"

**经过**：v0.7.12.0 把 BP 门槛从 ≥2 轮升到 ≥6 轮 + 覆盖 ≥4，是为了防空 BP。但完全没设计"用户被拦后的下一步路径"——只给了一个干巴巴的 422 错误文案。

**根因**：设计时把"门槛"当成"输出过滤器"，没把它当成"产品交互节点"。

**给未来的提醒**：
- ⚠️ **任何"拦截/限制/阈值"逻辑设计时必须配套设计"被拦后的引导路径"**
- ⚠️ "用户能干什么"和"用户被拦后能干什么"是两个完全不同的设计任务，不能省

#### 🟡 坑 2：JSX 字符串里嵌中文双引号导致 tsc 报错

**经过**：写 BPGateDialog 按钮文案时用了 `带"未充分会诊"水印`，tsc 直接报 `error TS1005: '}' expected.`

**修复**：改成中文方括号 `带「未充分会诊」水印`

**给未来的提醒**：
- ⚠️ JSX 字符串里**永远不要**用英文双引号 `"`（即使是中文内容）
- ⚠️ 引号嵌套优先用中文「」或单引号 `'`

#### 🟡 坑 3：track 事件没在 AnalyticsEvent 类型里加新事件类型直接用了

**经过**：写 Chat.tsx 用 `track("diagnosis_gate_blocked", ...)` 时忘了去 lib/analytics.ts 加类型定义，直接 tsc 三连报错。

**给未来的提醒**：
- ⚠️ **新埋点事件先去 lib/analytics.ts 加类型，再回到业务代码用**
- ⚠️ TypeScript 严格联合类型给的错是好事——不是它在阻拦，是你没遵守自己定义的契约

### 三、对的决策

#### ✅ 决策 1：把"消息注入"能力提到 ChatShell 而不是 Chat 自己管

**做法**：ChatShell 加 `injectAssistantMessage(content)` 函数 + 通过 prop 传给 Chat

**为什么对**：
- ChatShell 是 messages state 的 owner，注入逻辑应该在那
- Chat 只负责"我要注入一条消息"的意图，不管怎么实现
- 未来其他模块（比如客服弹窗、营销提示）都能复用这个能力

#### ✅ 决策 2：bridge API 必须有 LLM 兜底文案

**做法**：LLM 失败时返回 `fallback: string`，前端永远有内容可注入

**为什么对**：
- 如果 LLM 挂了 → 用户看到"网络错"→ 体验断裂
- 兜底文案虽然不如 LLM 灵活，但**能保住主流程不卡住**
- 三档差异化兜底文案 30 行代码搞定

#### ✅ 决策 3：force 路径不删门槛，加 forced 标记 + 水印

**做法**：用户点「硬要现在出」→ generate?force=true → 跳过门槛检查 → report.forced=true → 卡片顶部加金色水印

**为什么对**：
- 既给了用户「我自己有数」的逃生口
- 又用水印告诉用户「这份不能晒，是给你自己看的」
- 不破坏「值得晒的 BP」品牌资产

### 四、给未来的提醒（v0.7.12.0 14 条铁律新增）

15. **任何"门槛/拦截"逻辑必须配套"被拦后路径"** —— v0.7.12.0 留下的最大教训
16. **JSX 字符串永远不嵌英文双引号** —— 中文内容用「」或 `''`
17. **新埋点事件先改 type 再用** —— 别把 TypeScript 错误当阻挠

### 三、哪些决策事后看是对的（值得复用）

#### ✅ 决策 1：4 commit 分步推送（不一次大 commit）

| commit | 层 | 行数 |
|---|---|---|
| `0783486` | 1/4 数据层（类型 + 4 模块） | +586 |
| `5b63f06` | 2/4 聊天层（fire-and-forget + 门槛 + 吃账本） | +192 |
| `b7d96ec` | 3/4 闭环层（summary API + 小行 + 提示 + OG cache） | +187 |
| `c840bf1` | 4/4 修缮层（emoji + 版本号 + ROADMAP/README） | +221 |

**为什么对**：
- 每个 commit 内部一致（同一层的事在一起）
- 任意一个 commit 出问题可独立 revert，不污染其他层
- 给未来 git bisect 留好锚点

**给未来的复用**：
- 任何 ≥4h 的 patch 都应该拆成 3-4 个 commit，每个 commit 可独立体检

#### ✅ 决策 2：fire-and-forget 判官，不阻塞主对话

**为什么对**：
- 主对话流式回复用户感知 0 影响（`done` 事件已发，判官在后台跑）
- 任何一步失败（judge LLM / KV / saveLedger）都静默吞，绝不抛回主链路
- 性能预算 ~300ms 用户完全感知不到

**给未来的复用**：
- 任何"为后续功能埋数据"的逻辑都应该走 fire-and-forget，绝不阻塞用户主路径

#### ✅ 决策 3：hasUserFact 双重兜底（规则 + LLM）

**做法**：
- 用户消息**不含 hasUserFact** 时直接跳过判官调用（防虚高 + 省钱兜底）
- 命中 hasUserFact 才调判官 → 减少 50%+ 不必要的 LLM 调用

**为什么对**：
- 一个简单规则函数（数字 / 大写词 / 长度 ≥30 / 具体动词）解决了 80% 的"是否值得调判官"问题
- 月成本预估从 ¥1.4 降到 ¥0.7

**给未来的复用**：
- 任何"调 LLM 判定"的场景都应该先想"有没有规则版能筛一遍"

#### ✅ 决策 4：BP 门槛 422 + 结构化 gate 字段

**做法**：不达标返 422 + `gate: { missingRounds, missingCoverage, currentTurns, currentCoverage }`

**为什么对**：
- 前端可直接展示"还差 X 轮 + Y 块"具体进度，不是模糊"再聊聊"
- 422 状态码标准化（与限流的 429 区分），方便监控告警

**给未来的复用**：
- 业务规则错误一律返 422 + 结构化字段，不要 400/500 + 文字 error

#### ✅ 决策 5：双 repo 文档同步策略

**做法**：
- 主 repo 改 README/ROADMAP/CHANGELOG/LAUNCH_LOG/DEPLOY 5 个文档
- mind repo 改 README/_methodology_history.md 2 个文档
- 都用同一个版本指针 v0.7.12.0，时间戳一致

**为什么对**：
- 未来任何时候打开任意一个文档都能找到"当前在哪个版本 + 上下文"
- mind repo 的 history 留下"主 repo v0.7.12.0 不动 mind repo 的原因"——下次类似场景不用重新思考

**给未来的复用**：
- 每个里程碑版本都必须做"双 repo 文档校对"作为最后一步
